## Clarifications

- Q: 實際領取物資的流程是採用「誠實商店/自助式」，還是需要主管/管理員點擊「核准/發放」後才能拿取或扣除庫存？ → A: 需管理員核准遞交 (Option B)
- Q: 當多名員工同時對數量有限的同一物資送出申請時，系統應該在什麼時候阻擋或處理超額申請？ → A: 送出申請時檢查可用量 (Option A)
- Q: 當一線員工的所屬區域被管理員改劃分到其他區域時，該員工之前送出、但尚未被核准的「待核准」申請單應該如何處理？ → A: 自動退回/拒絕 (Option C)
- Q: 系統應該如何防止未授權人員進入並瀏覽庫存？ → A: 管理員後台審核開通 (Option A)
- Q: 當雲端儲存空間 (Firebase Storage) 已滿或圖片上傳失敗時，前端的預設顯示行為為何？ → A: 顯示文字警告與預設圖 (Option A)

### User Story 1 - 居服員 (一線人員) 透過通訊軟體快速註冊與領用 (Priority: P1)

居服員初次使用系統時，透過通訊軟體 (LINE) 登入並輸入姓名與所屬區域完成註冊。未來的領用操作中，系統會按其登入的 LINE 帳號自動帶入由管理員校準後的正式名稱與劃定的區域。**居服員只能進入並瀏覽自己所屬的分區頁面**，選擇或留言所需物資提交申請（此申請將進入「待審核」狀態）。等待對應區域的管理員核准後，系統方進行庫存扣除與正式發放。

**Why this priority**: 這是系統最核心的使用者旅程，確保物資能夠被順利申請並綁定正確的身分與區域。

**Independent Test**: 可以透過模擬新用戶登入、填寫資料，並且提交物資申請來獨立測試此流程。

**Acceptance Scenarios**:

1. **Given** 居服員尚未註冊，**When** 居服員透過通訊軟體登入並同意授權，**Then** 系統導向註冊頁面並要求輸入姓名與區域。
2. **Given** 居服員註冊完成但尚未開通，**When** 居服員登入系統，**Then** 系統不得顯示空白頁面，而是顯示包含明確「待審核圖標」與「待管理員審核開通」訊息的畫面，並隱藏所有庫存與物資資訊。
3. **Given** 有一筆待開通的新用戶設定，**When** 管理員在後台核對身分並核准開通，**Then** 該用戶下次登入時會自動帶入校準後的正式名稱與區域，並且 **只能進入自己所屬的分區物資申領頁面**。
4. **Given** 居服員在申領頁面，**When** 居服員選取物資並點擊送出，且該物資的「可用庫存（實際庫存 - 待核准數量）」大於等於申請量，**Then** 系統生成一筆「待核准」的申請單並發送電子郵件通知給對應區域的管理員。
5. **Given** 居服員在申領頁面選取物資，**When** 該物資的「可用庫存」小於申請數量時，**Then** 系統拒絕送出申請，並提示庫存不足。
6. **Given** 有一筆待核准的申請單，**When** 管理員在後台點擊審核並核准，**Then** 系統扣除實際庫存、將狀態更新為「已發放」，並記錄於異動日誌中。

### User Story 2 - 子管理員進行身分校準與分處管理 (Priority: P2)

子管理員登入後台後，**可以進入所有區域（如：新莊區、三蘆區、板中永區）**。子管理員可以檢視各區域的「初次註冊」人員，修改其「備註名稱」（例如改為員工編號或全名）並調整所屬區域。同時，子管理員可以修正各分處的庫存數量。

**Why this priority**: 確保資料的準確性與庫存的即時調整，是維持系統正常運作的重要管理機制。

**Independent Test**: 可藉由子管理員帳號登入系統，確認可跨區進入，編輯員工名單及修改庫存數據來驗證。

**Acceptance Scenarios**:

1. **Given** 子管理員進入後台，**When** 點擊進入任一區域的新註冊員工名單進行編輯，**Then** 可以修改該員工的備註名稱與區域並儲存。
2. **Given** 子管理員進入庫存管理，**When** 修改任一分處物品的庫存數量並輸入修正理由，**Then** 庫存數量更新，並在異動日誌中產生一筆修改紀錄。

---

### User Story 3 - 管理部經理的全域管理與警報機制 (Priority: P3)

管理部經理擁有最高權限，**可以進入所有區域**，可更換系統企業標誌、更新物資照片、修正全區庫存、設定安全存量，並查看全區的庫存異動日誌。此外，**管理部經理可以在未來自行新增「區域頁面」，用以擴充要管理的品項與對應的新分區**。系統在物品庫存低於安全存量時，會自動觸發警報並寄送電子郵件給經理與教育員。

**Why this priority**: 掌握全域概況並確保物資不會斷貨是高階管理者的核心需求，並且賦予系統彈性擴充的能力。

**Independent Test**: 透過經理帳號登入，測試圖片上傳、動態新增區域、跨區庫存調整及安全存量低於閾值時是否寄發通知。

**Acceptance Scenarios**:

1. **Given** 經理進入全域設定，**When** 上傳新的企業標誌與物資照片，**Then** 前台使用者的介面立即更新顯示新圖片。
2. **Given** 某物資的現有庫存量低於設定的安全存量，**When** 系統偵測到此狀態，**Then** 系統自動發送警告電子郵件給經理與教育員。
3. **Given** 經理進入區域管理，**When** 點擊新增區域並設定新區域名稱，**Then** 系統動態生成新區域頁面，允許管理員加入並管理新的品項，且居服員可在註冊時選擇該新區域。

### Edge Cases

- **員工區域異動處理**：當管理員更改某位員工的所屬區域時，系統將自動掃描並「拒絕/退回」該員工在原區域所有仍處於「待核准」狀態的申請單，並釋放佔用的可用庫存配置。員工需在新區域重新提交申請。
- **儲存空間/圖檔異常處理**：若雲端儲存空間異常或圖片無法載入，系統仍允許正常申請操作，但介面上該物品的圖片區域會呈現「圖片載入失敗，請核對文字名稱」之警告佔位圖，降低拿錯物資的風險。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 支援透過通訊軟體 (LINE) 進行居服員的單一登入 (SSO) 身分驗證。系統 MUST 按登入的 LINE 帳號劃定區域，且一般使用者（居服員）登入後 **只能進入自己所屬的分區**。初次註冊後帳號預設為「待開通」狀態，**登入時系統 MUST 提供明確的「待審核圖標與訊息」UI 畫面（避免空白頁）**，需經管理員核准後方可瀏覽與申請物資。
- **FR-002**: 系統 MUST 允許子管理員修改所屬區域員工的「備註名稱」與「所屬區域」。當員工所屬區域被修改時，系統 MUST 自動將該員工所有「待核准」的申請單狀態更新為「已拒絕/已取消」，並釋放對應的可用庫存。
- **FR-003**: 系統 MUST 維護「可用庫存（實際庫存 - 所有待核准數量）」，並在員工提交申請前，阻擋申請量超過此「可用庫存」的請求。
- **FR-004**: 系統 MUST 允許居服員提交物資領用申請，提交後申請單狀態預設為「待核准」。
- **FR-005**: 系統 MUST 在員工送出申領時，根據其所屬區域自動發送電子郵件通知給該區的管理員。
- **FR-006**: 系統 MUST 允許子管理員及經理針對「待核准」的申請單進行審核（如：核准、拒絕），且僅在「核准」後扣除實際庫存。若拒絕，則釋放該申請佔用的可用庫存配置。
- **FR-007**: 系統 MUST 支援多個區域的庫存與精確存放地點標註，且 **系統 MUST 提供介面讓最高權限管理員（經理）動態新增區域頁面**，以利未來增加要管理的品項與分區。
- **FR-008**: 系統 MUST 提供一個不可竄改的異動日誌，自動記錄時間、操作者(備註名)、品項、變動數與變動後餘額。
- **FR-009**: 系統 MUST 要求管理員在手動修正庫存時，強制輸入修正理由（如：盤損、過期）。
- **FR-010**: 系統 MUST 提供安全存量警報設定，並在低於安全量時自動發送電子郵件給專門負責人（經理與教育員）。
- **FR-011**: 系統 MUST 具備嚴格的分層角色權限控管：包含最高權限（管理部經理）、分處權限（子管理員）及一般權限（居服員）。**子管理員與管理部經理 MUST 能夠跨區進入所有區域（初始預設為：新莊區、三蘆區、板中永區，未來可由管理者更改名稱或動態新增）**。
- **FR-012**: 系統 MUST 實作基於 Firebase Firestore Rules 的資料庫層級安全性規則 (Security Rules)，嚴格阻擋未登入或狀態為「待開通/待審核」的使用者讀取任何庫存與區域資料；僅允許已核准的使用者依其角色權限讀取或修改對應範圍的資料。
- **FR-013**: 系統的 Firestore Rules 檔案 MUST 納入版本控制，並設定 CI/CD 流程 (如 GitHub Actions)，確保當規則檔在 GitHub 上更新時，能自動連動並部署更新至 Firebase 的 Firestore 規則中。

### Key Entities

- **使用者 (User / Care Worker / Admin)**: 包含 LINE 唯一識別碼、登入名稱、管理員校準後的備註名稱、所屬區域、角色權限（經理、子管理員、居服員）與帳號狀態（待開通、已核准、停權）。
- **區域 (Region / Zone)**: 包含區域 ID、區域名稱。此為可動態新增之實體，用以劃分物資與人員的歸屬。
- **物資與庫存 (Inventory Item)**: 包含物品名稱、照片網址、現有庫存量、安全存量閾值、所屬區域 (對應至特定區域實體) 與詳細存放位置（如：某某貨架）。
- **申請單 (Fulfillment Request)**: 包含申請人資訊、申請物資明細、申請時間、當前狀態（待核准、已發放、已拒絕）與審核人。
- **異動日誌 (Transaction Log)**: 包含紀錄時間、操作者（備註名）、異動品項、變動數量、變動後餘額與修正理由。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一線人員能於 1 分鐘內完成初次綁定註冊與物資申領操作。
- **SC-002**: 物資申領送出後，對應區域的管理員能於 3 分鐘內收到正確的電子郵件通知。
- **SC-003**: 當庫存低於安全存量時，系統能可靠地觸發並發送警報信件，送達率達 99% 以上。
- **SC-004**: 所有使用者的介面體驗在主流手機裝置上皆能正常顯示行動優先的版面與提供直覺的按鈕操作。
